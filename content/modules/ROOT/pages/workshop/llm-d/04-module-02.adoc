= Module 2: Observe Single-GPU Behaviour
:source-highlighter: rouge
:toc: macro
:toclevels: 2

In this module you'll look at the vLLM metrics endpoint, open the Grafana dashboard, and run a short illustrative benchmark to see how a single GPU handles high concurrency.
These results are for observation only—the comparison benchmarks run in Modules 3 and 4.

== Learning objectives

By the end of this module, you'll be able to:

* Locate vLLM Prometheus metrics and the Grafana dashboard
* Run a GuideLLM benchmark against a single-GPU deployment
* Observe queue saturation under concurrent load

== Exercise 1: Explore Grafana monitoring

To understand vLLM performance, you need metrics visualization.
We'll start by looking over the vLLM metrics themselves, then move on to a more visual representation.
There is a pre-deployed Grafana dashboard that will help with this lab.
Let's make sure it's up and recording data from our model server instance.

=== Explore vLLM metrics endpoint

vLLM exposes Prometheus metrics on its metrics port. Verify the service is exposing metrics:

[NOTE]
====
Your port-forward from Module 1 should still be running. If it's no longer active, restart it with:

`oc port-forward deployment/llama-vllm-single-predictor 8000:8000 &`
====

[.console-input]
[source,bash]
----
# Fetch metrics
curl http://localhost:8000/metrics | grep vllm | head -40
----

You should see a lot of vLLM-specific metrics, including:

* `vllm:num_requests_running`
* `vllm:num_requests_waiting`
* `vllm:kv_cache_usage_perc`
* `vllm:generation_tokens_total`

[.console-input]
[source,bash]
----

pkill -f "port-forward"
----

== Exercise 2: Run GuideLLM benchmark

Now establish your performance baseline by running GuideLLM with a KV-cache-aware prompts file.
Unlike synthetic random prompts, this dataset contains carefully constructed prompts sized to 80% of the model's KV cache capacity, with each prompt repeated multiple times.
This workload is designed to produce high cache hit rates when requests are routed intelligently—making it ideal for comparing vLLM versus llm-d.

=== Understand the benchmark data

Your lab environment has a pre-provisioned PVC (`benchmark-data`) containing a `prompts.csv` file.

Verify the PVC and data are available:

[.console-input]
[source,bash]
----
# Verify the benchmark-data PVC exists
oc get pvc benchmark-data

# Check the prompts file is present
oc run check-data --image=registry.access.redhat.com/ubi9/ubi-minimal:latest \
  --restart=Never --rm -it \
  --overrides='{"spec":{"volumes":[{"name":"data","persistentVolumeClaim":{"claimName":"benchmark-data"}}],"containers":[{"name":"check-data","image":"registry.access.redhat.com/ubi9/ubi-minimal:latest","command":["sh","-c","wc -l /data/prompts.csv && head -1 /data/prompts.csv"],"volumeMounts":[{"name":"data","mountPath":"/data"}]}]}}' \
  -- sh
----

You should see the number of rows in the CSV and the header line showing `prompt,output_tokens_count` columns.

=== Run the benchmark

This is a short, illustrative run—just 30 seconds per concurrency level—to show how a single GPU handles a large volume of concurrent requests.
We are not collecting these results for later comparison; that happens in Module 3 with 4 replicas running a full 300-second benchmark.
The goal here is simply to observe what happens when one GPU gets overloaded.

[.console-input]
[source,bash]
----
cat << 'EOF' | oc apply -f-
apiVersion: batch/v1
kind: Job
metadata:
  name: guidellm-vllm-single
spec:
  backoffLimit: 0
  ttlSecondsAfterFinished: 86400
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: guidellm
          image: ghcr.io/vllm-project/guidellm:latest
          env:
            - name: GUIDELLM_TARGET
              value: "http://llama-vllm-single-predictor:8000/v1"
            - name: GUIDELLM_PROFILE
              value: concurrent
            - name: GUIDELLM_RATE
              value: "32,64"
            - name: GUIDELLM_OUTPUTS
              value: json,csv
            - name: GUIDELLM_MAX_SECONDS
              value: "30"
            - name: GUIDELLM_DATA
              value: "/data/prompts.csv"
            - name: GUIDELLM_PROCESSOR
              value: "RedHatAI/Meta-Llama-3.1-8B-Instruct-FP8-dynamic"
            - name: HF_HOME
              value: /tmp/hf-cache
            - name: TRANSFORMERS_CACHE
              value: /tmp/hf-cache
            - name: XDG_CACHE_HOME
              value: /tmp
          volumeMounts:
            - name: cache
              mountPath: /tmp/hf-cache
            - name: results
              mountPath: /results
            - name: data
              mountPath: /data
              readOnly: true
      volumes:
        - name: cache
          emptyDir: {}
        - name: results
          emptyDir: {}
        - name: data
          persistentVolumeClaim:
            claimName: benchmark-data
EOF
----

[WARNING]
====
**This benchmark takes approximately 3 minutes to complete** (30 seconds per concurrency level).
====

Follow the benchmark logs:

[.console-input]
[source,bash]
----
oc wait --for=condition=ready pod -l job-name=guidellm-vllm-single --timeout=120s
oc logs -f job/guidellm-vllm-single
----

Press `CTRL+C` when the job completes to return to the command prompt.

[TIP]
====
**If your terminal disconnects**, you can resume monitoring with:
[source,bash]
----
oc logs -f job/guidellm-vllm-single
----
To check if the job has completed:
[source,bash]
----
oc get job guidellm-vllm-single
----
====

=== Monitor in Grafana

While the benchmark runs, open your Grafana dashboard and watch:

1. **TTFT metrics**: Watch how P95 latency climbs at concurrency 64 vs 32
2. **KV cache hit rate**: With random round-robin routing, cache hits are low even with repeated prompts
3. **GPU utilization**: The single GPU will saturate quickly at high concurrency

image::benchmark-dashboard-single.png[]

=== Observe the results

Focus on the **`ℹ Request Latency Statistics (Completed Requests)`** section—specifically the `TTFT` and `Request Latency` columns.
Both the median and p95 TTFT are very high: the single GPU is already deeply queued within the first 30 seconds.

Example output:

[.console-output]
[source,bash]
----
ℹ Request Latency Statistics (Completed Requests)
|============|=========|========|========|========|=======|=======|=======|=======|
| Benchmark  | Request Latency || TTFT           || ITL          || TPOT         ||
| Strategy   | Sec             || ms             || ms           || ms           ||
|            | Mdn     | p95    | Mdn    | p95    | Mdn   | p95   | Mdn   | p95   |
|------------|---------|--------|--------|--------|-------|-------|-------|-------|
| concurrent | 41.3    | 41.3   | 1691.2 | 1691.2 | 159.2 | 159.2 | 165.3 | 165.3 |
| concurrent | 30.3    | 30.3   | 284.5  | 284.8  | 120.5 | 120.5 | 121.1 | 121.1 |
|============|=========|========|========|========|=======|=======|=======|=======|
----

These numbers are erratic and very high—requests are queuing behind each other's long prefill computations.
We won't carry these forward for comparison. Module 3 scales to 4 replicas and runs a proper 300-second benchmark.
