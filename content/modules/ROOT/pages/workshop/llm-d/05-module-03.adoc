= Module 3: Scale vLLM & Establish Comparison Baseline
:source-highlighter: rouge
:toc: macro
:toclevels: 2

The obvious response to high latency is more GPUs.
In this module you'll scale to 4 replicas and run a proper 300-second GuideLLM benchmark.
The results you record here are the numbers you will compare against llm-d in Module 4.

== Learning objectives

By the end of this module, you'll be able to:

* Scale a KServe InferenceService to 4 replicas with random load balancing
* Run a statistically meaningful GuideLLM benchmark (300 seconds per concurrency level)
* Record P95 TTFT as the baseline for the llm-d comparison

== Exercise 1: Deploy vLLM with 4 replicas

You'll create a single InferenceService configured to scale to 4 replicas,
with each replica running on its own GPU.

=== Deploy scaled vLLM InferenceService

Create an InferenceService with 4 replicas for naive horizontal scaling:

[.console-input]
[source,bash]
----
cat << 'EOF' | oc apply -f-
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: llama-vllm-scaled
  annotations:
    openshift.io/display-name: Llama 3.1 8B FP8 Scaled
    security.opendatahub.io/enable-auth: "false"
    serving.kserve.io/deploymentMode: RawDeployment
    opendatahub.io/hardware-profile-name: accelerated-profile
    opendatahub.io/hardware-profile-namespace: redhat-ods-applications
spec:
  predictor:
    automountServiceAccountToken: false
    maxReplicas: 4
    minReplicas: 4
    deploymentStrategy:
      type: Recreate
    model:
      modelFormat:
        name: vLLM
      name: ''
      resources:
        limits:
          cpu: '4'
          memory: 8Gi
          nvidia.com/gpu: '1'
        requests:
          cpu: '4'
          memory: 8Gi
          nvidia.com/gpu: '1'
      runtime: rhaiis-cuda
      storageUri: 'oci://registry.redhat.io/rhelai1/modelcar-llama-3-1-8b-instruct-fp8-dynamic:1.5'
EOF
----

Expected output:

=== Load balancing setup

KServe creates a headless service (`clusterIP: None`) for the predictor, which pins each connection to a single pod.
For benchmarking we need a standard ClusterIP service that distributes requests across all 4 replicas.

Create one now:

[.console-input]
[source,bash]
----
cat << 'EOF' | oc apply -f-
apiVersion: v1
kind: Service
metadata:
  name: llama-vllm-scaled-lb
spec:
  type: ClusterIP
  selector:
    serving.kserve.io/inferenceservice: llama-vllm-scaled
  ports:
    - port: 8000
      targetPort: 8000
EOF
----

Expected output:

[.console-output]
[source,bash]
----
service/llama-vllm-scaled-lb created
----

=== Run the benchmark

[WARNING]
====
**This benchmark takes approximately 10 minutes to complete** (5 minutes per concurrency level).
You may want to press `Enter` occasionally to prevent the Showroom environment from timing out.
====

[.console-input]
[source,bash]
----
cat << 'EOF' | oc apply -f-
apiVersion: batch/v1
kind: Job
metadata:
  name: guidellm-vllm-scaled
spec:
  backoffLimit: 0
  ttlSecondsAfterFinished: 86400
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: guidellm
          image: ghcr.io/vllm-project/guidellm:latest
          env:
            - name: GUIDELLM_TARGET
              value: "http://llama-vllm-scaled-lb:8000/v1"
            - name: GUIDELLM_PROFILE
              value: concurrent
            - name: GUIDELLM_RATE
              value: "32,64"
            - name: GUIDELLM_OUTPUTS
              value: json,csv
            - name: GUIDELLM_MAX_SECONDS
              value: "300"
            - name: GUIDELLM_DATA
              value: "/data/prompts.csv"
            - name: GUIDELLM_PROCESSOR
              value: "RedHatAI/Meta-Llama-3.1-8B-Instruct-FP8-dynamic"
            - name: HF_HOME
              value: /tmp/hf-cache
            - name: TRANSFORMERS_CACHE
              value: /tmp/hf-cache
            - name: XDG_CACHE_HOME
              value: /tmp
          volumeMounts:
            - name: cache
              mountPath: /tmp/hf-cache
            - name: results
              mountPath: /results
            - name: data
              mountPath: /data
              readOnly: true
      volumes:
        - name: cache
          emptyDir: {}
        - name: results
          emptyDir: {}
        - name: data
          persistentVolumeClaim:
            claimName: benchmark-data
EOF

oc wait --for=condition=ready pod -l job-name=guidellm-vllm-scaled --timeout=120s
oc logs -f job/guidellm-vllm-scaled
----

Press `CTRL+C` when the job completes to return to the command prompt.

[TIP]
====
**If your terminal disconnects**, you can resume monitoring with:
[source,bash]
----
oc logs -f job/guidellm-vllm-scaled
----
To check if the job has completed:
[source,bash]
----
oc get job guidellm-vllm-scaled
----
====

== Exercise 3: Record your results

Note the **TTFT p95** values for concurrency 32 and 64 from your run—you will compare these directly against Module 4.

The p95 TTFT at concurrency 64 will be high: many seconds.
Throughput is 4x higher than a single GPU, but tail latency has barely moved.
The prompts file was designed to be cacheable—but with 4 replicas and round-robin routing, there is only a 1-in-4 chance of hitting the right replica's cache.

The hardware is there. The routing intelligence is not. That changes in Module 4.

== Module 3 summary

**What you accomplished:**

* Scaled vLLM to 4 GPUs with a standard ClusterIP load balancer
* Ran a full 300-second GuideLLM benchmark
* Recorded baseline P95 TTFT numbers for the Module 4 comparison

**Key takeaway:**

More GPUs increase capacity, but isolated KV caches and random routing keep P95/P99 TTFT high.
The prompts are cacheable—the routing just can't exploit that.
